User Story

During the last sprint of the project, I was mainly responsible of implementing one user story, because I was also the scrum master. The story implied added functionality that would enable the end user to filter available vets from unavailable vets on the web interface.

Have an endpoint to filter available from unavailable vets

As a user, I want to filter out vets that are enabled and disabled so I know with who shall I take an appointment for my pet.

Acceptance criteria:
• When the switch is on, the Vet is available.
• When the switch is off, the vet is unavailable.
• Available and unavailable vets will be displayed using different endpoints.

This story had proper realistic, as well as clear and specific acceptance criteria, which made the implementation clearer, and also it had story points estimates that were thoughtfully attributed by our team.

Test-Driven Development

As Unit testing was a crucial part of this project, I used the JUnit 5 library in order to successfully build application unit tests. More specifically, I ensured coverage of the implementation for my Endpoint Filtering story. However, looking back at the project, I think that additional front-end Selenium tests could have been done, but this was not planned, especially because of the lack of time (It was also not our responsability, we were not in charge of the front-end but mostly the API Gateway.). Since my user story for this sprint is pretty small, I managed to reach at least 90% of testing of the code coverage in my unit tests, and also offer great readability thanks to the built-in Java Docs documentation that I added at each step of my tests to ensure great readability by other collaborators.

Code

Since my responsibility is mostly related to the API gateway service, most of our operations happen in the domain client and in the presentation layers of the Pet Clinic project.

Throughout the code that I have developed for my user stories, my code was meeting the requirements set in the acceptance criteria, since my methods were taking the same arguments and attributes as defined in the requirements, and the response of the code implemented were exactly the results expected in the story description and acceptance criteria. For example, for the deleteVet user story, it was necessary that we pass the Id of the vet we want to delete in the URL, and that we in return delete the vet associated to the selected ID from the database. Effectively, our code implementation uses the Id of the Vet in the REST endpoint URL, and in return deletes that vet from our DB.

The code implemented for each of my user stories can also be used in other languages, as the attributes used are universal and bilingual, and the code is well documented, which would make it easy to add a translation to the exact URL of the request. I also made used of the DTOs in our API Gateway service in order to perform operations on the data.

To comply with the data model, I used the REST Controller in the Presentation Layer, the Service Client in the domain client layer, as well as the DTOs in the data layer for every functionality that I implemented.

Security

As far as security is concerned, since my endpoint enables the user to select available and unavailable vets, there is not a huge security requirement to consider through the implementation. That said, however, I made sure to use DTOs where possible and needed to ensure efficient and secure structure.

Also, I ensured that all the direct database requests that my code performed on the SQL container were built with query parameters, which greatly reduces the risk for injection vulnerabilities. Lastly, I kept the code simple with well-documented Java Docs so that it would be easy to maintain for security purposes.

Code Review

Regarding the review aspect of this sprint, I used techniques to make sure that my code could be well reviewed. First of all, I added embedded Java Docs documentation across all of my code to be reviewed, so that the reviewers could easily understand, interpret and review my code. Second of all, I made sure to assign reviewers for my Pull Request that were in my team, since they are well-involved with the API Gateway structure, and I chose reviewers that had worked on similar User stories.

After my colleagues had reviewed my code, I then took note of their recommendations, and determined that I needed to refine the implementation of my unit tests, as well as to validate my implementation directly with the VETS team as they were working on our story at the same time as we were.

For this sprint, I did not get assigned as a code reviewer for any user stories, which did not give me the opportunity to review others' code. Lastly, I made sure to comply with Jira and GitHub by following the naming conventions and rules set forth in the repository's documentation.

End-to-End Testing

Since the Selenium tests implementation was not completely ready, we were not able to develop Selenium tests for this Sprint. Also, since our service is the API-Gateway, the majority of our implementation is done on the backend and no graphical UI is involved in our User Stories.

Deployment

The story that I implemented in the last sprint of this project was basically an addition to the current implementation of the Vets service. That said, it was built so that they it could be deployed using continuous integration and development (CI/CD) procedures. Firstly, since each service in this project is structured using Docker, the entirety of my implementation for this sprint is containerized inside the API-Gateway Docker container. I also made sure to utilize a domain-driven design approach, which means that all of the different sections of the code for my endpoint filter story resides in the appropriate separate application layers. This greatly facilitates the workflow and maintenance in a CI/CI development environment.
