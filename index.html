User Story

Throughout the initial sprint of the project, my team and I had difficulties with the user stories we were assigned to. The problems were stopping us from moving forward, as we were encountering many bugs, execution errors and exceptions. However, we took the opportunity of Sprint 2 to move forward in the project, and I decided to adopt a proactive approach. I talked with my team, and we decided that I would push my user stories of Sprint 1 to Sprint 2. Those user stories that I had to implement were the following:

Delete Vet

As a user, I want to remove Vets that do not work at the pet clinic anymore.

Acceptance criteria:
• An HTTP DELETE request must be sent when the id of the target record is specified in the URL.
• If the DELETE request is successful, the HTTP response must display a confirmation page that the record has been deleted.

Update Vet

As a user, I want to be able to update information about vets to keep the information up-to-date.

Acceptance Criteria:
• The HTTP PUT request must be sent when the id of the record is in the URL.
• The HTTP response must show a form with fields to update the record information.

The stories both had proper realistic acceptance criteria, which made the implementation clearer, and also respectively had story points estimates that were thoughtfully attributed by our team. 

Also, as I was working on implementing the DeleteVet and doing my tests, I figured that the getVet (method for getting single vet) was not done, as well as the CreateVet method. I needed those for my tests, so I decided to use an Agile approach and be proactive. I therefore decided to reach out to my teammate which was responsible of this story. After informing him, I took over his two stories as well, and therefore successfully implemented three stories for Sprint 2, which are createVet, deleteVet and getVet(single).



Version Control

During Sprint 2, I utilized Version Control techniques in order to work seamlessly in sync with the shared GitHub repository on the project. Firstly, I created my own branch that followed the naming convention. The branch I initially created was for the first story I was working on, which was DeleteVet.

I also made sure to commit at different intervals, basically I committed every time my code had something new and working to it. I also left clear commit messages for each one. However, as I took over Vlad’s two stories, I did the changes and implementation in the code all at once, and therefore decided to put the new changes for createVet, deleteVet and getVet on the same branch. When I did this, I ensured to combine the commit messages into one, so that the documentation would stay clear and consistent for better readability.

Afterwards, I finally created a pull request, using the appropriate naming template, in order for two of my teammates to review my code and to accept the PR and merge it to the main branch.

Test-Driven Development

I adopted the concept of Test-Driven Development for my Create, Delete and Get (CDG) user stories for this Sprint. In fact, I made use of the JUnit 5 library in order to successfully build application unit tests. More specifically, I ensured coverage of all implementation for my CDG stories, since I tested my Delete implementation by having a combined test that firstly creates a vet using the Create method, then retrieves the created vet using the Get method, then deletes the vet using the Delete method, and then verifies if the vet still exists with the Get method again. All together, the combined unit tests that I built provide at least 90% of testing of the code coverage, and also offer great readability thanks to the built-in Java Docs documentation that I added at each step of my tests to ensure great readability by other collaborators.

Regarding the test data used, I used as mentioned above the method to Create, Delete and Get a Vet, and chose real-world test data to make my test as relevant as possible to the real-world.

Code

Since my responsibility is mostly related to the API gateway service, most of our operations happen in the domain client and in the presentation layers of the Pet Clinic project.

Throughout the code that I have developed for my user stories, my code was meeting the requirements set in the acceptance criteria, since my methods were taking the same arguments and attributes as defined in the requirements, and the response of the code implemented were exactly the results expected in the story description and acceptance criteria. For example, for the deleteVet user story, it was necessary that we pass the Id of the vet we want to delete in the URL, and that we in return delete the vet associated to the selected ID from the database. Effectively, our code implementation uses the Id of the Vet in the REST endpoint URL, and in return deletes that vet from our DB.

The code implemented for each of my user stories can also be used in other languages, as the attributes used are universal and bilingual, and the code is well documented, which would make it easy to add a translation to the exact URL of the request. I also made used of the DTOs in our API Gateway service in order to perform operations on the data.

To comply with the data model, I used the REST Controller in the Presentation Layer, the Service Client in the domain client layer, as well as the DTOs in the data layer for every functionality that I implemented.

Security

Since we are part of the API Gateway service, we are more involved with direct API queries. That said, in Sprint 2, I made sure to use DTOs everywhere it was needed in order to have a securely and efficiently structured implementation. In order to utilize secure programming techniques, I made sure that all the direct database requests that my code performed on the SQL container were built with query parameters, which greatly reduces the risk for injection vulnerabilities. Also, I kept the code simple with well-documented Java Docs so that it would be easy to maintain for security purposes.

Code Review

Regarding the review aspect of this sprint, I used techniques to make sure that my code could be well reviewed. First of all, I added embedded Java Docs documentation across all of my code to be reviewed, so that the reviewers could easily understand, interpret and review my code. Second of all, I made sure to assign reviewers for my Pull Request that were in my team, since they are well-involved with the API Gateway structure, and I chose reviewers that had worked on similar User stories.

After my colleagues had reviewed my code, I then took note of their recommendations, and determined that I needed to refine the naming of my unit tests, as well as to separate my tests into different testing scenarios.

For this sprint, I did not get assigned as a code reviewer for any user stories, which did not give me the opportunity to review others' code.

Lastly, I made sure to comply with Jira and GitHub by following the naming conventions and rules set forth in the repository's documentation.

End-to-End Testing

Since the Selenium tests implementation was not completely ready, we were not able to develop Selenium tests for this Sprint. Also, since our service is the API-Gateway, the majority of our implementation is done on the backend and no graphical UI is involved in our User Stories.

Deployment

The stories that I implemented in Sprint 2 were built so that they could be deployed using continuous integration and development (CI/CD) procedures. Firstly, since each service in this project is structured using Docker, the entirety of my implementation for this sprint is containerized inside the API-Gateway Docker container. I also made sure to utilize a domain-driven design approach, which means that all of the different sections of the code for my user stories resides in the appropriate separate application layers. This greatly facilitates the workflow and maintenance in a CI/CI development environment. Also, as shown below, all of my Unit tests successfully pass and the code successfully builds.

Scrum and Agile Adherence

During Sprint 2, I attended most of the scrum meetings with my teammates so that everyone would stay connected and aligned on what to do. I also reached out many times to my team partners with issues that I was facing with the implementation of my code, to get their feedback and outside opinion. Also, when participating in the scrum meetings, I brought to the table what I had done up to that moment, what was next for me to implement, as well as the elements that were slowing me down or restricting me from going forward.

I kept a proactive approach during the whole sprint. As opposed to the first sprint of this project, I used an Agile approach to make sure that I stayed in control over the whole process. To do so, I focused on self-organization and team collaboration, but I put a big emphasis on value-based prioritization. This is a big change from Sprint 1, where I kept being stuck on my first user story because I had a hard time with exceptions and code errors. In Sprint 2, I proactively started the implementation of my user stories. If I was stuck somewhere, I would reach out to my teammates for help, while at the same time evaluating if starting another user story and coming back at the issue later on was valuable based on the situation. Also, as mentioned previously, when I was building my tests, if one of my teammates’ user story was not ready yet but was needed for me to complete my story, I would talk with the person assigned to that story and agree to take over his user story. This allowed me to move forward faster while getting stuck way less.

I was also there to do troubleshooting and testing to help my teammates in the completion of their user stories. Team collaboration is really important in an Agile environment, and in the next sprint, I will pay even more importance to the daily scrum meetings to make sure that the whole team stays 100% on track.
